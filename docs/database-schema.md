# Database Schema & Data Models

This document outlines the main entities used across the application and how they map to our Firebase Firestore collections.

## Global Rules
- **Security:** Finance documents live below `/users/{uid}`. Rules require the authenticated UID to match that path and validate allowed fields, primitive types, enum values, date shape, string sizes, and installment bounds.
- **References:** Documents typically store foreign keys as strings (e.g., `userId`, `plannedExpenseId`).

## Collections

### 1. `users/{uid}/transactions`
Represents individual financial movements (both income and expenses).

**Document Structure:**
```typescript
interface Transaction {
  id?: string;               // Automatically generated by Firestore
  description: string;
  amount: number;            // Stored as float/number
  paymentMethod: string;     // e.g., 'Crédito', 'Débito', 'Pix'
  installments: number;      // 1 for single payment
  date: string;              // ISO format: YYYY-MM-DD
  type?: 'income' | 'expense';
  category?: string;         // E.g., 'food', 'transport', 'salary'
  plannedExpenseId?: string; // (Optional) Link to the planned expense that generated this
  sourceKey?: string;        // Deterministic idempotency key for generated transactions
}
```

### 2. `users/{uid}/plannedExpenses`
Represents future or recurring commitments that aren't realized transactions yet.

**Document Structure:**
```typescript
interface PlannedExpense {
  id?: string;
  userId?: string;           // Legacy optional owner field; ownership comes from the path
  description: string;
  amount: number;
  dueDate: string;           // ISO format: YYYY-MM-DD
  isRecurring: boolean;
  recurrenceInterval: number;// in months (e.g., 1 for every month)
  status: 'pending' | 'confirmed' | 'cancelled';
  type?: 'income' | 'expense';
  category?: string;
  paymentMethod?: string;
  installments?: number;
  sourcePlannedExpenseId?: string; // Previous recurrence in the deterministic chain
}
```

### 3. `creditCards` (Planned/Implemented)
Represents the user's credit cards to track limits and invoices.

**Document Structure:**
```typescript
interface CreditCard {
  id?: string;
  userId: string;
  name: string;              // e.g., 'Nubank', 'Itaú'
  limit: number;             // Total card limit
  closingDay: number;        // Day of the month the invoice closes
  dueDay: number;            // Day of the month the invoice is due
}
```

## Data Flow Notes
- Confirming or rejecting a pending planned expense runs in a Firestore transaction. Generated transaction and recurrence IDs are deterministic, so concurrent retries converge instead of creating duplicates.
- Backup imports validate the complete payload before writing and commit at most 400 operations per batch. Supplied IDs are rejected when they would overwrite an existing document.
- Monetary values intentionally remain in the decimal `amount` field for this scalability iteration; an integer-cent migration is deferred.
